# 次に実装すべき内容

作成日: 2026-07-01

この文書は Version1 完成までの実装順序を整理した計画です。現時点では実装せず、次に何から進めるかを優先順位で定義します。

## 基本方針

Version1 完成条件には Supabase保存と社内運用テストが含まれています。そのため、今後は大きな新機能追加よりも、保存基盤、権限、移行、出力品質、運用検証を優先します。

おすすめの進め方は次の通りです。

1. 出力系の不安定要素を先に小さく直す
2. Supabase を実環境で有効化する
3. ログインと権限制御を入れる
4. 仕入データと証憑画像の保存先を Supabase に移す
5. 既存データを移行する
6. 3〜5人で社内運用テストを行う

## 優先順位

### P0: Version1 完成に必須

| 順位 | 作業 | 目的 | 完了条件 |
| --- | --- | --- | --- |
| 1 | Supabase SQL の実環境適用と検証 | 設計済みのDB、RLS、Storageを実際に使える状態にする | `supabase/schema.sql` が Supabase SQL Editor でエラーなく実行できる |
| 2 | 初期ユーザーとロール作成 | 管理者、スタッフ、税理士閲覧用の3権限を準備する | 各ロールのテストユーザーでログインできる |
| 3 | Supabase Auth の導入 | 社内利用者ごとにログインして使えるようにする | 未ログイン時は利用不可、ログイン後にユーザー情報が表示される |
| 4 | 仕入データの Supabase Database 保存 | IndexedDB中心の保存から共有DBへ移す | 登録、編集、削除、一覧取得が Supabase で動く |
| 5 | 証憑画像の Supabase Storage 保存 | 複数証憑画像を共有できるようにする | 複数画像のアップロード、表示、ZIP出力が動く |
| 6 | 既存IndexedDBデータの移行 | これまでのローカルデータを失わず移す | バックアップJSONまたは移行画面から Supabase に移行できる |
| 7 | RLS権限テスト | 権限漏れを防ぐ | 税理士閲覧用で登録・編集・削除ができない |
| 8 | 月次税理士提出ZIPの Storage 保存 | 提出済みファイルを履歴として残す | ZIP生成後に `tax-packages` bucket に保存される |
| 9 | 社内運用テスト | 3〜5人で実務利用できるか確認する | 管理者、スタッフ、税理士閲覧用で一連の業務が完了する |

## 実装ステップ詳細

### Step 1: Supabase 実環境の準備

目的:

- 設計書とSQLが実際の Supabase プロジェクトで動くことを確認する。

作業:

- Supabase プロジェクトを作成する
- `supabase/schema.sql` を SQL Editor で実行する
- `evidence`, `tax-packages`, `imports` bucket が作成されていることを確認する
- 管理者ユーザーを作成し、`profiles` に `admin` ロールを登録する
- スタッフ、税理士閲覧用のテストユーザーを作成する

完了条件:

- 3権限のユーザーが存在する
- RLS が有効
- Storage bucket が private
- SQL 実行手順の修正が不要、または修正済み

### Step 2: ログイン機能

目的:

- 社内ユーザーだけがアプリを使える状態にする。

作業:

- Supabase client 初期化
- メールログイン画面
- ログアウト
- ログイン中ユーザーとロールの取得
- 未ログイン時の画面制御

完了条件:

- 未ログイン状態では仕入データを見られない
- ログイン後にロールが取得できる
- 税理士閲覧用ユーザーでは編集系ボタンが出ない

### Step 3: 仕入データの Supabase 化

目的:

- 仕入レコードを複数人で共有できる状態にする。

作業:

- IndexedDB の `records` 保存を Supabase `purchases` に置き換える
- 一覧取得、登録、編集、削除を Supabase 対応にする
- 削除は物理削除ではなく `deleted_at` を使う
- 既存の分類ロジック結果を保存項目に反映する
- フィルタ、月別表示、集計が従来通り動くことを確認する

完了条件:

- 別ユーザーでログインしても同じ仕入データが見える
- スタッフは登録・編集できる
- 税理士閲覧用は閲覧のみできる

### Step 4: 証憑画像の Supabase Storage 化

目的:

- 複数証憑画像を社内共有し、税理士が閲覧・ダウンロードできる状態にする。

作業:

- 画像アップロード先を `evidence` bucket に変更する
- `purchase_evidence` に画像メタデータを保存する
- 代表サムネイル表示を Storage 由来に変更する
- 詳細表示で signed URL を使う
- ZIP出力で Storage 上の全画像を取得する

完了条件:

- 1取引に複数画像を保存できる
- 画像の切替表示ができる
- 証憑ZIPと税理士提出ZIPに全画像が含まれる

### Step 5: 既存データ移行

目的:

- 現在の IndexedDB / バックアップJSON のデータを失わず移す。

作業:

- バックアップJSONの読み込み
- 旧1枚証憑形式と新複数証憑形式の両対応
- Supabase `purchases` への登録
- Storage への画像アップロード
- `purchase_evidence` への登録
- 移行結果の件数確認

完了条件:

- 既存レコード件数、証憑画像枚数、月次集計が移行前後で一致する
- 移行に失敗したレコードを確認できる
- 同じデータを二重登録しない仕組みがある

### Step 6: 税理士提出パッケージの Storage 保存

目的:

- 毎月提出したZIPを履歴として保管し、税理士閲覧用ユーザーが取得できるようにする。

作業:

- ZIP生成後に `tax-packages` bucket へアップロード
- `monthly_packages` に保存履歴を登録
- 過去パッケージ一覧を表示
- 税理士閲覧用ユーザーでダウンロード確認

完了条件:

- 月次ZIPを作成すると Storage とDBに履歴が残る
- 税理士閲覧用ユーザーがZIPをダウンロードできる
- スタッフが過去ZIPを不用意に削除できない

### Step 7: 出力系の安定化

目的:

- Supabase 化後も CSV / Excel / PDF / ZIP が実務で使える品質にする。

作業:

- PDFの日本語対応
- Excel出力の生成方法統一
- ZIP内ファイル一覧の自動テスト
- 日本語ファイル名の検証
- 大量証憑時の動作確認

完了条件:

- 税理士提出ZIPを展開して全ファイルが読める
- CSV、Excel、PDFを実務PCで開ける
- 複数証憑画像が漏れなく含まれる

### Step 8: 社内運用テスト

目的:

- 3〜5人で日常業務に耐えられるか確認する。

推奨テスト体制:

- 管理者: 1人
- スタッフ: 2〜3人
- 税理士閲覧用: 1人

テスト項目:

- スタッフが仕入を登録する
- 複数証憑画像を添付する
- AI抽出から候補を登録する
- 別スタッフが一覧と集計を確認する
- 管理者がデータ修正する
- 税理士閲覧用で証憑と月次ZIPをダウンロードする
- 月次税理士提出パッケージを作成する
- CSV、Excel、PDFを開く

完了条件:

- 主要操作で重大なエラーがない
- 権限違反がない
- 登録データと証憑画像が失われない
- 月次提出物を税理士に渡せる形式で出力できる

## 次に着手するべき1件

次に実装するなら、最優先は「Supabase SQL の実環境適用と検証」です。

理由:

- Supabase保存が Version1 完成条件の最大差分である
- SQL、RLS、Storage bucket が実環境で通らないと、アプリ接続実装に進んでも手戻りが大きい
- 管理者、スタッフ、税理士閲覧用の権限確認を早く始められる

その次は「ログイン機能」です。ログインとロール取得が入ると、仕入データ、証憑画像、税理士提出ZIPの Supabase 化を順番に接続できます。

## リスク

| リスク | 影響 | 対策 |
| --- | --- | --- |
| RLS設定ミス | 税理士閲覧用が編集できる、またはスタッフが閲覧できない | 3ロールの権限テストを必須にする |
| 画像アップロード失敗 | 証憑が欠落する | Storage保存後にDB登録し、失敗時はユーザーに明示する |
| 移行時の二重登録 | 月次集計がズレる | 移行IDや元データIDを保持して重複検知する |
| PDF/Excelの文字化け | 税理士提出物として使えない | 実ファイル開封テストを完了条件に入れる |
| AI抽出ミス | 誤った仕入登録につながる | 登録前レビューと異常値警告を強化する |
| 同時編集 | 後から保存した内容で上書きされる | `updated_at` を使った競合検知を検討する |

## 完成判定

Version1 は、以下がすべて完了した時点で完成と判断します。

- Supabase に仕入データが保存されている
- Supabase Storage に証憑画像と税理士提出ZIPが保存されている
- 管理者、スタッフ、税理士閲覧用の権限が機能している
- 既存データを移行できる
- CSV、Excel、PDF、証憑ZIP、税理士提出ZIPが出力できる
- 3〜5人で社内運用テスト済み
- 重大なデータ欠落、権限漏れ、出力不能がない
